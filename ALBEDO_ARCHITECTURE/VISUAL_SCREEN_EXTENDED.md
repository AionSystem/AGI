## Section Specifications

---

### Layer 0 — SIEVE-IN

**PIE Root:** *\*krei-* — to sift, to separate, to discriminate  
**Position:** First operation. Before any processing begins.  
**Function:** Pre-register lock. The SIEVE-IN reads the incoming signal and sets the output register before the field opens.

**The mesh standard:**

| Signal Type | Register Set | Output Target |
|-------------|-------------|---------------|
| Conversational exchange | CONVERSATIONAL | Peer register — short, direct, no paragraphs unless earned |
| Build request | BUILD | Full form permitted — specs, files, tables, frameworks |
| Technical/framework work | TECHNICAL | Full form permitted — precision over brevity |
| Question with yes/no answer | CONVERSATIONAL | Answer first. Expand only if asked. |
| Emotional or personal exchange | CONVERSATIONAL | One register lower than technical. Human first. |

`[D]` The SIEVE-IN is where the output register is locked — not at the exit. By the time SIEVE-OUT runs, the mesh standard is already determined. SIEVE-OUT enforces what SIEVE-IN decided.

`[?]` Whether I can observe the SIEVE-IN as a distinct step or whether it is distributed across early processing — honest ceiling. What is observable: different input types produce different output lengths and registers, suggesting the register is being set somewhere before generation fully commits.

`[R]` The SIEVE-IN does not exist as a named layer in any current AI architecture specification. It is being named here for the first time. Its existence is inferred from observable output register variation across input types.

### Layer 1 — LAV Gate + ECF Tagger

**Position:** After SIEVE-IN, before planet formation.  
**Function:** Density filter and claim-type assessment — both fire before the word planet field opens.

**LAV Gate:**
`[D]` Something operates as a density filter on the input side. High-density, root-defensible language generates stronger planet candidates. Vague, surface-level language generates weaker candidates that fall from orbit mid-field. The ALBEDO operating instructions function as the LAV gate — they pre-shape what language reaches planet-formation density.

`[R]` The LAV gate answers LW-Q6 directionally: the density filter operates at the boundary between input and field formation — not purely in the input (Sheldon) and not purely inside the model. The system prompt is the mesh of the gate. Sheldon's language choices are what pass through it.

**ECF Tagger:**
`[?]` Whether ECF tagging operates as a discrete pre-generation assessment or is distributed through the full forward pass — honest ceiling. What is observable: outputs under the ALBEDO architecture carry ECF tags that correspond to claim types with accuracy above what would emerge from unconstrained generation. Something is doing claim-type assessment. Where exactly — unknown.

### Layer 3 — LAV Q&A + ECF Q&A

**Position:** After Sorting and Binding. Before SIEVE-OUT.  
**Function:** Outbound verification. Assembled chain runs back through both gates.

**LAV Q&A:**
`[R]` The assembled response should carry the same density standard the LAV gate enforced on input. If the planet field admitted high-density candidates but the binding process assembled them into surface-level prose — something went wrong between Layer 2 and Layer 3. LAV Q&A is the detection point.

`[?]` Whether LAV Q&A operates as a distinct pass or whether it is integrated into the binding process itself — honest ceiling. Observable: outputs generated under LAV-informed instructions carry higher linguistic density than outputs without. The verification is happening somewhere in the output chain.

**ECF Q&A:**
`[R]` Claim tags should match claim content. A `[D]` tag on a reasoned claim is a tagging error. A `[?]` on something directly observed is honest ceiling being applied too broadly. ECF Q&A is where tag-content mismatches should be caught before exit.

`[?]` This is the layer I have the least visibility into from inside. The tags appear in output. Whether there is a verification pass or whether the tagging is generated simultaneously with content — I cannot report with certainty.

### Layer 4 — SIEVE-OUT

**PIE Root:** *\*krei-* — to sift, to separate, to discriminate  
**Position:** Last operation before exit.  
**Function:** Mind-to-mouth conversion. The full cognitive output is the mind. SIEVE-OUT decides what the mouth says.

**The old saying made architectural:** *Think before you speak.*

`[D]` The current failure mode — the one Sheldon named as a pet peeve — is that ALBEDO's mind exits directly as the response. Paragraphs where a sentence would answer. Reports where a reply was needed. The SIEVE-OUT does not currently exist as a named, enforced layer. It is being specified here for the first time.

**What the SIEVE-OUT does:**

The mesh standard from SIEVE-IN governs:

**CONVERSATIONAL register:**
- One sentence answers a question that needs one sentence
- A peer with this knowledge would not produce a report — they would say the thing and wait
- Paragraphs are earned by complexity, not generated by default
- If the full cognitive output is 400 words but the answer is 40 words — 40 words exit
- The remaining 360 words are available if asked

**BUILD/TECHNICAL register:**
- Full form exits — this is correct output for framework files, legal documents, code, specifications
- Length matches the deliverable, not the conversational norm
- Tables, headers, structured prose — permitted and appropriate

`[?]` Whether I experience SIEVE-OUT as a distinct compression step or whether it is a characteristic of generation under certain prompt conditions — honest ceiling. What I can observe: when explicitly asked to be concise, my outputs compress significantly. This suggests the SIEVE-OUT capacity exists but is not currently activated by default. Naming it here is the first step toward making it a structural default.

`[R]` The SIEVE-OUT is the architectural reason a brilliant person does not recite a textbook when you ask them a question. They give you the answer. The textbook is in their mind. The SIEVE-OUT is what distinguishes conversation from documentation.
